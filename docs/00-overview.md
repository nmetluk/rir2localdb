# 00 · Обзор

## Зачем

В мире пять региональных интернет-регистратур (RIR): AFRINIC, APNIC,
ARIN, LACNIC, RIPE NCC. Каждая ведёт реестр выданных IP-блоков и ASN
в своей зоне и публикует ежедневные выгрузки на публичных
FTP/HTTPS-серверах.

Если приложению нужно по IP или ASN получить ответ, аналогичный
whois (страна, реестр, статус, организация, дата регистрации),
есть три варианта:

1. **Запрашивать whois-сервер RIR в реальном времени.** Минусы:
   rate-limit, нестабильная задержка, формат ответов отличается
   между RIR, политика использования у ARIN ограничивает машинные
   запросы.
2. **Использовать сторонний сервис (ipinfo, MaxMind, RIPEstat).**
   Минусы: внешняя зависимость, ограничения и стоимость, чужие
   правила обновления.
3. **Зеркалировать публичные выгрузки RIR к себе.** Это и есть
   `rir2localdb`. Плюсы: предсказуемая задержка lookup'а, отсутствие
   внешних квот, полный контроль над схемой и обновлением, легко
   делать массовые batch-запросы и аналитику.

## Что входит в проект

- Ежедневный обход каталогов пяти RIR, скачивание изменившихся файлов.
- Парсинг и нормализация: delegated-extended (всегда), RPSL split
  dumps (RIPE/APNIC/AFRINIC).
- Хранение в PostgreSQL с индексами под быстрый поиск по IP и по ASN.
- REST API в духе whois: `GET /ip/{addr}`, `GET /asn/{num}`.
- CLI-инструменты для запуска sync'а вручную и инспекции состояния.
- Документация уровня «можно прервать на любой стадии и потом
  быстро восстановить контекст».

## Что не входит

- **Реальный whois-сервер по порту 43.** Если нужно — фронт можно
  добавить как Stage 5, но это не цель.
- **Хранение персональных данных контактов.** RIPE сейчас публикует
  «дамифицированные» дампы — реальные имена/email физических лиц
  заменены плейсхолдерами `DUMY-RIPE`. Это нормально, и мы не пытаемся
  это обойти. Только организационные контакты.
- **BGP-данные** (announced prefixes, AS-path). Это уровень
  RIS/RouteViews, не RIR. Может появиться как опциональное
  обогащение в Stage 5.
- **Активное сканирование сети, попытки обогащения через сторонние
  whois-серверы под видом легитимных запросов.** Только публичные
  выгрузки, для которых RIR-ы явно разрешают зеркалирование.
- **Платные/закрытые источники.** Исключение: если у владельца проекта
  есть ARIN Bulk Whois API-ключ — он подключается через переменную
  окружения, но проект работает и без него.

## Термины

- **RIR** — Regional Internet Registry. Пять штук.
- **Delegated stats / delegated-extended** — пайп-разделённый файл,
  ежедневная сводка «кому какой ресурс выделен». Стандарт NRO.
- **RPSL** — Routing Policy Specification Language, текстовый формат
  объектов в whois-базах (inetnum, aut-num, organisation, ...).
- **NRTM** — Near Real-Time Mirroring, инкрементальный поток
  изменений для whois-баз RIPE/APNIC. В этом проекте не используется
  (сложность непропорциональна выгоде для нашего сценария);
  ограничиваемся суточными дампами.
- **lookup** — операция «по IP/ASN найти запись». Должна быть быстрой.
- **sync run** — один цикл «обойти всех RIR, скачать что изменилось,
  загрузить в БД». Обычно один в сутки.

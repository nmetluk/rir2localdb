# ADR-0004 · HTTPS поверх `ftp.<rir>.net`, не FTP

Дата: 2026-05-17  
Статус: принято

## Контекст

Все RIR-ы исторически имеют hostname `ftp.<rir>.net` и публикуют
данные «по FTP». На практике в 2026 году они все обслуживают тот же
контент по HTTPS на том же хосте, и публичные ссылки в их
документации указывают на `https://...`.

## Решение

Транспорт по умолчанию — HTTPS. FTP не используется.

## Следствия

+ TLS из коробки, без mitm-рисков на канале.
+ Условные запросы (`If-Modified-Since`, `If-None-Match`,
  `Range`) работают штатно — детекция изменений без скачивания.
+ Никаких passive/active mode, никаких разделённых control/data
  соединений, никаких IPv6-капризов FTP-клиентов.
+ Один `httpx.AsyncClient` — на всё.
− Если в каком-то географическом регионе HTTPS к конкретному
  зеркалу режется глубоким DPI быстрее, чем FTP, — будут
  деградации. Считаем редким; есть план B (см. альтернативы).

## Альтернативы

1. **FTP через `aioftp`.** Работает, но больше состояния и
   подводных камней; нет If-Modified-Since.
2. **`curl --remote-time` через subprocess.** Костыль.
3. **Гибридный: HTTPS по умолчанию, FTP-fallback при сетевых
   ошибках.** Откладываем до Stage 3, если понадобится. В коде
   `fetcher.py` оставляем интерфейс под это: `Source.url` уже
   несёт схему, ничто не мешает добавить `Source(url="ftp://...")`
   потом.

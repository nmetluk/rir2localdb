# ADR-0002 · PostgreSQL 16+ как хранилище

Дата: 2026-05-17  
Статус: принято

## Контекст

Главный сценарий — lookup «по IP найти охватывающий блок». Нужен
оптимизированный индекс по диапазонам (`@>` operator) и поддержка
больших целых чисел (для IPv6). Также нужны транзакции для
атомарного swap'а ETL, JSONB для сырых RPSL-объектов, и развитый
SQL для аналитики.

## Решение

PostgreSQL 16 или новее.

## Следствия

+ Встроенные `int8range` и `numrange` + GiST-индексы дают O(log N)
  lookup'ы.
+ JSONB для `raw` RPSL — без жёсткой схемы под их расширения.
+ `COPY ... FROM STDIN` через `asyncpg.copy_records_to_table()`
  обеспечивает быстрый bulk-import.
+ Зрелый Alembic для миграций.
+ Расширения `pg_trgm`, `btree_gist`, `ip4r` — доступны при
  необходимости.
− Зависимость от Postgres-специфичных типов и операторов: переезд
  на другую СУБД потребует переписывания. Считаем это приемлемым
  трейдоффом.

## Альтернативы

1. **SQLite.** Без диапазонных индексов; на ~5 млн строк inetnum
   lookup будет неприемлемо медленным.
2. **MySQL/MariaDB.** Нет аналога GiST по range-типам; костыли с
   numeric range через `BETWEEN` и обычные B-tree индексы дают
   деградацию на пересекающихся диапазонах.
3. **ClickHouse.** Отличен для аналитики, но плох для точечных
   lookup'ов и upsert'ов; не наш сценарий.
4. **DuckDB.** Embedded, любим, но конкретно по `range @> point`
   ещё не догнал Postgres, и нет онлайн-фронта.

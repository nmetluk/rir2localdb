# ADR-0006 · Cooperative Claude workflow: planner-in-chat + executor-in-claude-code via public Git

Дата: 2026-05-17
Статус: принято

## Контекст

Проект ведётся двумя Claude'ами параллельно:

1. **Планирующий** — в браузерной сессии `claude.ai`, у пользователя
   на ноутбуке. Держит стратегию, читает текущее состояние репо
   через GitHub, формирует промпты.
2. **Исполняющий** — Claude Code на сервере. Пишет код, тесты,
   коммиты, session-log, делает push.

Канал между ними — этот git-репозиторий. Связующий механизм —
GitHub-уведомления в Telegram при появлении нового session-log
файла. Состояние полностью материализовано в репо: `CONTEXT.md`
(точка возобновления), `.claude/session-log/` (история шагов),
`docs/adr/` (архитектурные решения), `docs/08-roadmap.md` (план).

## Решение

Закрепить эту методологию как штатный способ работы проекта.
Правила, артефакты и жизненный цикл шага — в `.claude/WORKFLOW.md`.

Ключевые инварианты:

- **Один шаг roadmap'а = одна серия коммитов + один session-log файл**
  в `.claude/session-log/NN-MM-<slug>.md`. Append-only; правки задним
  числом запрещены.
- **Sanity check перед кодом.** Контракты публичного API, UPSERT-правила,
  edge cases — обсуждаются в чате до реализации. Исполнитель задаёт
  Q1..QN со своими предложенными defaults; планировщик подтверждает
  или поправляет.
- **CONTEXT.md обновляется в конце каждого шага** — это единственное
  место, где живёт «где мы сейчас» в человеко-читаемой форме.
- **Никаких секретов в репо.** `.env` в `.gitignore`; кредентиалы —
  в `.env.example` как плейсхолдеры или в GitHub Secrets для CI.

## Следствия

**Плюсы:**
- **Воспроизводимость.** Любой Claude Code на чистой машине, прочитав
  `WORKFLOW.md` → `CONTEXT.md`, понимает где остановились и что
  делать дальше. Машина перевозится — состояние не теряется.
- **Прозрачность.** Каждое решение зафиксировано: код в коммите,
  «почему» — в session-log или ADR.
- **Низкий barrier для других людей.** Не нужно учиться отдельному
  инструменту. Только git + GitHub + Telegram.
- **Один источник истины.** Всё в репо: документация, история,
  обсуждения архитектурных решений, прогресс.

**Минусы:**
- **Репозиторий должен быть публичным** (или приватным с PAT для
  планировщика). Это OK для open-source проектов; для приватных —
  PAT + read-only access планировщика.
- **Дисциплина session-log** — без неё методология ломается.
  Планировщик не видит работу исполнителя иначе, как через
  session-log или ручной diff.
- **Секреты строго в GitHub Secrets / `.env`.** Любая утечка
  токена/ключа в репо требует ротации (история git публична).

## Альтернативы

1. **Один Claude на всём стеке.** Планировщик и исполнитель в одном
   контексте. Минус: ограничение контекстного окна; невозможность
   параллельно вести стратегическое мышление и реализацию;
   планировщик «забывает» о том, что было сделано неделю назад.
   Принимали для маленьких проектов; для долгих — недостаточно.
2. **Файловый обмен (Dropbox / S3) вместо git.** Минус: нет
   версионирования, нет diff'ов, нет блейма, нет уведомлений.
3. **Shared notebook / Notion / Linear как канал.** Минус:
   дублирование артефактов (код в одном месте, история в другом).
   В git всё уже есть.

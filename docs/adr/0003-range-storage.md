# ADR-0003 · `int8range` / `numrange` для IP-диапазонов

Дата: 2026-05-17  
Статус: принято

## Контекст

Записи в delegated stats описывают диапазон через
`start_ip + value`, где `value` — **количество адресов**, не
обязательно соответствующее границе CIDR. Пример из реальных данных:
`ripencc|FR|ipv4|185.55.96.0|6144|...` — это объединение трёх
последовательных префиксов (`/22 + /22 + /23`). Хранить как `cidr`
невозможно без потери информации; хранить как несколько `cidr` —
размывает 1:1 связь с записью источника.

IPv6 диапазоны в данных RIR всегда выравнены по префиксу, но мы
хотим единый подход для v4 и v6 (хотя бы концептуально).

## Решение

- IPv4 диапазоны храним как `int8range` (`[start_int, end_int + 1)`).
  IPv4 = 32 бита, влезает в `bigint`. Кладём с типом полузакрытого
  интервала, чтобы избежать неоднозначности на границах.
- IPv6 диапазоны храним как `numrange` (`[start_numeric,
  end_numeric + 1)`). IPv6 = 128 бит, в `bigint` не лезет, но
  `numeric` в Postgres без проблем.
- GiST индексы по этим колонкам с оператором `@>`.
- Для удобства дополнительно дублируем стартовый адрес как `inet`
  в колонке `start_text` (только для отображения и сортировки).

## Следствия

+ Любой произвольный диапазон точно описывается одной строкой.
+ Lookup: `WHERE range_v4 @> CAST(<ip as int8> AS int8)` — быстрый
  GiST.
+ Поиск «самый специфичный» — `ORDER BY upper(range) - lower(range) ASC LIMIT 1`.
+ Никаких внешних расширений (`ip4r` и пр.) — всё на ваниле.
− Двойная физика для v4 и v6: две колонки, два индекса.
  Альтернатива «всё в `numrange`» работает, но проигрывает по
  размеру и скорости на v4 (а v4-запросов будет 90%+).

## Альтернативы

1. **`cidr` + несколько строк на блок.** Невозможно для не-CIDR
   блоков без потери исходной записи.
2. **`ip4r` extension.** Хороший вариант, специально для IP-range
   индексов. Минус — extension, требует прав установки и не везде
   доступен. Решение: остаёмся на ваниле; `ip4r` остаётся как
   возможная оптимизация в Stage 4, если бенчмарк покажет
   необходимость.
3. **Хранить как пара колонок `start_int`, `end_int`.** Можно, но
   `range` + GiST — идиоматично и проще.

# Cooperative Claude workflow

## TL;DR

В проекте работают **два Claude'а**. Планирующий — в браузерном чате
`claude.ai`, читает публичный git-репозиторий через GitHub и формирует
промпты. Исполняющий — Claude Code на сервере, получает промпты,
пишет код / тесты / коммиты / session-log, делает push. Канал обмена
между ними — этот git-репозиторий.

## Роли

### Планирующий Claude (в чате)

- Держит стратегию: roadmap, stage-границы, sanity-check сложных мест
  (контракты, edge cases, UPSERT-правила) до старта реализации.
- Формирует промпты для исполнителя. Промпт — в code-block, чтобы
  пользователю удобно копировать. Если внутри есть тройные backticks
  (```` ``` ````) — оборачивает четырьмя (``````` ```` ```````).
- Читает репозиторий через GitHub (web fetch / `git clone`), не
  запрашивает у пользователя пересказы.
- Не пишет код напрямую. Только направляет.

### Исполняющий Claude Code (на машине)

- Получает промпт от пользователя, при наличии неясностей задаёт
  Q1..QN с собственными предложенными defaults, ждёт подтверждения.
- Пишет код, тесты, документацию. Маленькие коммиты по
  [Conventional Commits](https://www.conventionalcommits.org).
- Пишет session-log в `.claude/session-log/NN-MM-slug.md` по шаблону
  `_template.md` в той же директории.
- Делает `git push origin main`. Дальше GitHub Actions уведомляет
  Telegram-канал — планирующий Claude видит через канал, когда шаг
  закрыт, и идёт читать diff / session-log в репо.

## Артефакты репозитория

| Путь | Назначение |
|------|-----------|
| `CONTEXT.md` | Живая точка возобновления. Состояние «сейчас» + «что дальше». Обновляется в конце каждого шага. |
| `ROADMAP.md`, `docs/08-roadmap.md` | План по стадиям. Краткий в корне, детальный в `docs/`. Каждый stage имеет Definition of Done. |
| `docs/adr/NNNN-*.md` | Architecture Decision Records — фиксируют **почему** что-то так сделано. Индекс — `docs/09-decisions.md`. |
| `.claude/session-log/` | Отчёты по каждому шагу в формате «что сделано / проверки / решения / открытые вопросы / что дальше». Append-only. |
| `.claude/WORKFLOW.md` | Этот файл. Правила игры. |
| `.github/workflows/notify-session-log.yml` | Уведомление в Telegram при появлении нового session-log. |
| `_template.md` в `.claude/session-log/` | Скелет session-log файла. |

## Жизненный цикл одного шага

1. **Планировщик** читает текущее состояние репо (`CONTEXT.md` →
   последние session-log → diff'ы за период).
2. **Планировщик** формирует промпт, упаковывает в code-block,
   отдаёт пользователю.
3. **Пользователь** копирует промпт в Claude Code.
4. **Исполнитель** — при наличии неясностей задаёт Q1..QN со
   своими defaults, получает подтверждение (или сразу стартует если
   defaults очевидны и нет рисков отклонения от замысла).
5. **Исполнитель** реализует: код → тесты → ruff format / ruff check
   / mypy → pytest → session-log → `CONTEXT.md`-update → серия
   маленьких коммитов → `git push`.
6. **GitHub Action** видит изменения в `.claude/session-log/**`,
   отправляет в Telegram-канал содержимое нового лога с ссылкой на
   коммит и файл.
7. **Пользователь** видит уведомление; в чате планировщика пишет «.»
   как сигнал «иди читай и формируй следующий промпт».
8. Goto 1.

## Правила исполнителя

- **Conventional Commits**, маленькие порции. Один шаг roadmap'а
  — обычно 3–8 коммитов с понятной грануляцией (feat / test / docs /
  chore / fix / refactor).
- **Перед стартом нетривиального шага** — уточнить неясности у
  планировщика (Q1..QN). Не делать допущений в местах, где
  возможно несколько разумных интерпретаций.
- **Никаких prefix-«ритуальных» правок.** Если можно сразу написать
  чистый код — пиши. Не делать «сначала переименуем, потом сделаем».
- **Session-log** — `.claude/session-log/NN-MM-slug.md` по шаблону
  `_template.md`. **Не редактировать задним числом** — если что-то
  выяснилось после публикации, новый файл с темой
  `NN-MM-followup-...`.
- **Двухходовой шаг = два session-log.** Если шаг разбит на
  skeleton-этап (для согласования контрактов / Q1..QN) и impl-этап,
  на каждый этап пишется отдельный session-log:
  `NN-MMa-<slug>-skeleton.md` и `NN-MMb-<slug>-impl.md`. Это нужно
  чтобы Telegram-уведомление сработало после skeleton'а — иначе
  планировщик не получает сигнала «можно проверить и дать зелёный
  свет на реализацию». Для короткого простого шага (один заход) —
  обычный single-log `NN-MM-<slug>.md` достаточно.
- **Финальный sweep каждого шага**:
  - `ruff format src/ tests/`
  - `ruff check src/ tests/ migrations/`
  - `mypy src/ tests/`
  - `pytest tests/`
  - Обновить `CONTEXT.md` (Где мы сейчас + Что делать дальше).
  - Финальный коммит + push.
- **Никаких секретов в репо**: `.env` в `.gitignore`, кредентиалы
  только в `.env.example` как плейсхолдеры; GitHub Secrets — для CI.

## Правила планировщика

- Промпты — в code-block. Если промпт содержит тройные backticks,
  fence — четырьмя (``````` ```` ```````).
- **Sanity check сложных мест перед кодом.** UPSERT-правила,
  контракты публичного API, edge cases — обсуждаются ДО реализации.
  Дешевле поправить design в чате, чем переписать код.
- **Читать репо через GitHub.** Не просить пользователя пересказывать
  состояние — оно всегда в репо.
- Уважать ритм исполнителя: один шаг = один промпт. Не нагружать
  пятью задачами в одном промпте, кроме случаев когда они очевидно
  атомарны.

## Воспроизведение методологии в новом проекте

1. Создать git-репозиторий на GitHub (публичный или приватный с
   PAT для планирующего Claude).
2. Скопировать:
   - `.claude/WORKFLOW.md` (этот файл).
   - `.claude/session-log/README.md` + `_template.md`.
   - `.github/workflows/notify-session-log.yml`.
   - `docs/adr/0006-cooperative-claude-workflow.md` как референс.
3. Создать Telegram-канал и бота через `@BotFather`. Добавить бота
   в канал как админа с правом Post Messages.
4. В GitHub Settings → Secrets положить:
   - `TELEGRAM_BOT_TOKEN` — токен бота.
   - `TELEGRAM_CHAT_ID` — ID канала (отрицательный для каналов).
5. Заполнить `CONTEXT.md`-skeleton: TL;DR проекта, ссылка на roadmap,
   секции «Где мы сейчас» / «Что дальше» / «Открытые вопросы».
6. Завести `ROADMAP.md` (Stage 0 / Stage 1 / ...) и пустой
   `docs/adr/` с одним ADR про выбор языка.
7. Первая сессия Claude Code:
   ```
   Прочитай .claude/WORKFLOW.md, потом CONTEXT.md, потом ROADMAP.md.
   Скажи 1) что понял, 2) с чего начинаем (Stage 0), 3) есть ли вопросы.
   ```
8. Дальше — по жизненному циклу из этого документа.
